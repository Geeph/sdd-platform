## 0. 已定决策

沿用 M1–M4 已定且不再论证的约定：Node 24 LTS + TS strict、pnpm/tsup/vitest/oclif/biome、
provenance 只认 GitHub PR/merge 元数据且按明确 PR 号 / merge SHA 定位、fail closed、不建仓内
账本、gate check 由平台仓集中托管的 required workflows 产出、CLI 双层模式（CI 编排命令 +
业务逻辑在 factory 包）、M2 D8 check 名冻结、M2 D10 防伪边界、M4 D2 的"裁决类 vs 被裁决类
job"区分、M4 D8 的 `contract_changed` 窄口径定义。M4.5 新增决策：

- **D1 — `Contract Gate` 是 `ci-gate.yml` 里的一个顶层 job，job name 冻结为
  `Contract Gate`**。三个候选形态被排除，各有一票否决的理由：
  1. *独立 workflow 文件*（`contract-gate.yml`）：M2 的专用 organization ruleset 按
     `repository_id + path + sha` 逐文件登记 required workflow——新增文件意味着 org ruleset
     要登记第三个条目、`--finalize-protection` 的证据语义要扩展、每个存量产品仓要多一次
     迁移。放在 `ci-gate.yml` 内则自动继承 M2 已建立、M4 已复用的**同一枚 pin**（m4 D1
     的同仓库相对引用论证同样适用于同文件内的 job），零新增信任锚点。
  2. *`workflow_call` 被调 workflow 内部的 job*：GitHub 对 reusable workflow 内部 job 产生
     的 check run 命名为 `<caller job name> / <inner job name>`——不等于字符串
     `Contract Gate`，M1 `verifyGateApproval` 按精确名查找 evidence 会永远失败。M4 的四个
     平台 job 可以走 `workflow_call` 是因为**没有任何消费者依赖它们的 check 名**；
     Contract Gate 恰恰相反，check 名就是它对 M1/M5 的契约。
  3. *matrix job*：check 名带 `(matrix 值)` 后缀，同样破坏精确匹配。
  结论：单一顶层 job（job key `contract`，`name: Contract Gate`），内部**顺序**执行
  lint → diff → 三语言生成+编译+最小测试。M2 的 job 名守卫测试扩展到第三个冻结名（§7）。
- **D2 — 触发与门控**：`contract` job 带 `needs: [detect]` 与
  `if: needs.detect.outputs.contract_changed == 'true'`。`contract_changed` 的定义**原样
  消费 m4 D8/§2.7，不重新定义**：精确匹配 `contracts/openapi.yaml` 单一路径、
  `status !== 'removed'`（renamed 计入）；宽口径 `contracts/**` 平台矩阵信号与本 job 无关，
  两个字段继续各管各的（m4 D8 已论证不可混用，本文不复述）。`detect` 自身失败时
  `outputs.contract_changed` 是空字符串 → 本 job `skipped`——此时 `CI Gate` 由 m4 D13 的
  detect-first 短路直接失败，不依赖本文的聚合规则兜底。
- **D3 — `CI Gate` 聚合新增一段 contract 规则，键控 `contract_changed` 而非平台布尔**
  （m4 §5.2 预告的那次扩展，形状对齐 m4 D12 的两条规则）：
  1. `needs.contract.result` ∈ {`failure`, `cancelled`} → **无条件失败**（不看
     `contract_changed`）；
  2. `contract_changed == 'true'` 且 `needs.contract.result != 'success'` → **失败**。
     注意第二条刻意写 `!= 'success'` 而不是 `== 'skipped'`——plan §M4.5 的原文要求覆盖
     "skipped / failure / cancelled **或没有在当前 head SHA 产生结果**"四种情况，
     `!= 'success'` 一条全包（`failure`/`cancelled` 与规则 1 重叠是无害冗余，宁重不漏）。
  3. 其余情况（`contract_changed=false` 且 `skipped`；`true` 且 `success`；`false` 且
     意外 `success`）→ 通过。**自查（对照 m4 D12 的等价性论证方式）**：plan §M4.5 的判定
     要求逐条落位——`true+skipped`→规则 2；`true+failure/cancelled`→规则 1；`true+无结果`
     →在 job 级不可能出现"无结果"（`needs.*.result` 恒有值），它对应的真实场景是"check run
     没落在当前 head SHA 上"，那发生在 workflow 未触发的层面，由 required workflow 机制
     （无 check → `CI Gate` 本身 pending → 不可合并）覆盖，不是聚合逻辑可见的状态；
     `false+skipped`→通过（唯一允许 skipped 的情形）。`CI Gate` 的 `needs:` 列表加入
     `contract`。
- **D4 — Contract Gate job 不 checkout 产品仓：`contracts/` 子树经 API 读取（head 侧 +
  merge-base 侧），写入临时目录后对其运行钉死的工具**。理由：
  1. 它需要的产品内容只有 `contracts/**`（小体量、结构已知），Contents API 逐层 walk
     即可完整取回，不需要整仓 checkout；
  2. 这使 M4.5 **完全不依赖 m4 D24**（默认 `GITHUB_TOKEN` 能否 checkout 产品仓——那项验证
     属于 M4 的平台 job，至今未做）；API 读产品仓内容的能力已被 `pr-hygiene.yml` 在生产
     验证过；
  3. 与 M2 D10"平台 gate workflow 不 checkout 产品 PR blob"的表述保持字面一致，少一处
     需要解释的例外。
  **安全边界（评审修正）**：本 job 会编译并运行由不可信 `openapi.yaml` 派生的生成代码，
  因而 API 读取与生成/编译/测试必须是两个不同 workflow step：第一步持只读
  `GITHUB_TOKEN` 生成只含 PR 元数据与 `contracts/` 字节的快照；第二步不注入任何 token，
  只消费快照并运行工具链。`ToolRunner` 额外使用环境变量 allowlist，显式剔除
  `GITHUB_TOKEN`/`GH_TOKEN`/`ACTIONS_*` 等凭据变量，防未来重构把凭据重新继承给子进程。
  因此安全论证不是“token 只读所以可暴露”，而是“不可信代码执行域里根本没有 token”。
  Contract Gate 仍属于 m4 D2 的被裁决对象侧，使用 `pull_request` 而非
  `pull_request_target`，并受 D15 的资源上限约束。
- **D5 — 工具链选型与钉死（回应 plan §0"M4.5 前确定"，本文即定案；具体 patch 版本实现时
  按当时最新稳定核实，选型与钉死机制不可变——m3 D1 的同款约定）**：
  | 检查项 | 工具 | 钉死机制 |
  |---|---|---|
  | OpenAPI lint + 结构/`$ref`/examples/operationId 规则 | **spectral**（`@stoplight/spectral-core` + `spectral-rulesets`，程序内调用，不 shell CLI） | 平台仓 workspace devDependency + `pnpm-lock.yaml`（frozen install）；规则文件是平台仓资产（§2.3），产品 PR 不可改 |
  | breaking-change diff | **oasdiff**（Go 二进制） | workflow 步骤按固定版本从 GitHub releases 下载 + **sha256 校验**（不用第三方 action）；CLI 运行前核对 `oasdiff --version` 与内嵌期望值 |
  | TS / Swift / Kotlin client 生成 | **openapi-generator-cli**（单一 jar，generators：`typescript-fetch` / `swift5` / `kotlin`） | 固定版本 jar 从 Maven Central 下载 + sha256 校验；三份 generator config 是平台仓资产（固定包/模块名，§2.4） |
  | TS 编译 + 最小测试 | `tsc --noEmit` + vitest | 平台 workspace 既有 toolchain（Node 24 + lockfile） |
  | Kotlin 编译 + 最小测试 | Gradle（JDK 21 via `actions/setup-java`，与 M3/M4 锁定一致） | Gradle 固定版本（`gradle/actions/setup-gradle` 按完整 SHA pin，或直接 tarball+checksum，实现时择一）；**不假定生成项目自带 wrapper、也不信任其自带的构建工具版本声明**——一律用平台安装的固定版 Gradle 构建 |
  | Swift 编译 + 最小测试 | Swift 6 toolchain（swift.org ubuntu tarball + checksum） | **待决 #1**：`swift5` generator 产物在 Linux（FoundationNetworking）的可编译性需实现前验证；验证失败的 fallback = 整个 job 换 `runs-on: macos-14`（macOS 可跑全部三种语言；job 仅在合同变化时运行，macOS 分钟数代价可控——对照 m4 D14 对 macOS runner 的态度） |
  选 openapi-generator-cli 覆盖三语言而不是三个语言原生生成器（如 apple/swift-openapi-generator）：
  一个工具、一个版本、一次钉死、一份升级路径；某语言未来换原生生成器属于实现细节优化，
  不改变 gate 的对外契约（"三类 client 生成 + 编译 + 最小测试"）。
- **D6 — 六条 fail 条件（手册 §8.1 五项的可执行化，全部 error 级，任一命中 job 失败）**：
  1. **结构合法**：spectral `spectral:oas` 规则集 error 级零违例；且 `openapi` 字段必须
     匹配 `^3\.0\.\d+$`——**3.1 暂不支持**（openapi-generator 对 3.1 支持不完整、oasdiff
     的 3.1 行为需单独验证；这是显式的范围决策而非疏漏，放宽路径见 §12 待决 #2）。
  2. **operationId 完整性**：每个 operation 必有 `operationId`、全局唯一、匹配 M2 已有的
     `OPERATION_ID_RE`（`^[a-z][a-zA-Z0-9_-]*$`，复用同一常量不复制）。**跨里程碑一致性
     （自查）**：m4 D20 的 `diffOpenApiOperations` 对缺失/重复 operationId fail closed——
     本条在 gate 时挡住不合规合同，保证已合并合同永远不会让 M4 的 detect/impact 在
     operationId 上失败；两处用同一正则、同一语义，不会分道扬镳。
  3. **`$ref` 完整性 + 安全**：全部 `$ref` 可解析；只允许文档内引用（`#/...`）或
     `contracts/` 内相对路径；禁止任何 URL scheme、绝对路径、POSIX normalize 后逃逸
     `contracts/` 的 `..`（安全动机：远程 `$ref` = 构建时 SSRF + 不可复现输入；这也是
     D4"读 `contracts/` 子树即完备"成立的前提——本条保证合同的全部输入都在这个子树里）。
  4. **examples 完整性**：每个 operation 的 `requestBody` 与全部 2xx response 的每个
     media type 至少有 `example` 或 `examples` 之一（spectral 自定义规则）。范围定为
     "requestBody + 2xx"是本文对手册"examples 完整性"一词的具体化：错误响应通常复用
     `components` 且示例价值低——若评审认为过松/过严，调整的是规则文件一处常量（§12
     待决 #3），不动架构。
  5. **breaking diff**（见 D7）。
  6. **三语言 client 生成、编译、最小测试**：`typescript-fetch`/`swift5`/`kotlin` 三个
     generator 各自生成到临时目录 → 注入平台仓维护的**每语言最小 harness**（实例化
     client 入口对象/类型的一个测试文件；generator config 固定包/模块名使 harness 可
     编译，见 §2.4）→ 编译 + 运行。任一步非零退出 → fail。生成物**用后即弃**，绝不
     提交回产品仓（生成 client 入库属于实现 Issue 流程，见 §11）。
- **D7 — breaking 政策：检测到即默认失败，唯一放行路径是当前 Contract PR 中受版本控制的
  migration plan 显式确认；
  确认放行也永远伴随完整的 breaking 清单输出（§12.14 的 M4.5 半边："不静默"）**：
  - oasdiff 以 **merge-base 侧** `openapi.yaml` 为 base、PR head 侧为 head（merge-base
    语义见 D13），`ERR` 级发现视为 breaking。
  - base 侧不存在 `openapi.yaml`（首次引入）→ 无 diff 可做，本项跳过（首次合同不可能
    breaking），其余五条照常执行——"新增与修改同规则"（plan §M4.5）指的是**触发**同规则，
    不是给不存在的旧版本硬造一个 diff。
  - **base 侧存在但无法解析、或版本不受支持**（自查抓到的遗留死锁：M4.5 落地**之前**经
    Architecture PR 合入的旧合同没有过过本 gate，可能是 3.1、可能结构不合规）→ **视同
    "存在未知 breaking"**：走与 breaking 相同的受版本控制 migration plan 放行路径（报告标注
    "base contract unparseable, diff unavailable"），否则 fail。不这样处理的话，遗留
    产品的**每一个**后续合同 PR 都会在 diff 步骤 exit 3 且没有任何合法修复路径（改好
    head 也救不了 base）——fail closed 变成永久死锁；ack 路径把决定权交给人并留下审计
    痕迹，方向仍是保守的。
  - 检测到 breaking：PR body marker 的 `migration_plan` 必须匹配
    `^contracts/migrations/<marker.version>/[a-z0-9][a-z0-9-]*\.md$`；POSIX normalize 后仍须
    位于该目录。该文件必须在**当前 PR changed files** 中以 added/modified 出现、在 head
    快照中存在，并包含机读块 `sdd:breaking-change`，其中 `acknowledged: true`，正文去除
    机读块后非空。四项同时成立才放行；否则 fail。Issue/PR 号可写在 migration 文件正文中，
    但不能替代这个受版本控制的计划文件。
  - **为什么确认必须进 commit（评审修正）**：PR body 可在 CODEOWNER approval 之后编辑，
    不改变 head SHA，也不会触发 stale-review dismissal；因此 body 只能作为发现入口，不能
    作为已获人工批准的事实。migration 文件位于 `/contracts/`、属于当前 Contract PR 的
    head，任何新增或修改都会让 CODEOWNER approval 绑定到承载确认的最终 commit。
  - 无 breaking 但 marker 指向 migration plan → warning（报告提示“计划悬空”），不 fail——
    避免后来修掉 breaking 反而必须再次改 commit 的反直觉死锁。
  - §8.3 的"删除旧接口单独走 PR/Gate"落在同一机制里：删除 operation 的专门 Contract PR
    会被 oasdiff 判 breaking → 走同一条 migration 文件路径，正文记录迁移完成的证据。
  - **整文件删除 `contracts/openapi.yaml` → hygiene 直接 fail**（不是 Contract Gate 的
    职责——m4 D8 已定 `status='removed'` 不触发 `contract_changed`，gate job 不会跑；
    若无人拦截，"删掉整个合同"反而成为唯一不经任何合同检查的合同变更，这是明确的
    fail-open）。手册从未描述过整文件删除的合法流程，本文按保守原则把它挡在 hygiene
    （§4.3），需要时再设计专门流程。
- **D8 — 不支持"同一 PR 双 Gate"；合同变更（含首次引入）必须经专属 `gate:contract` PR——
  这是本文对手册 §6.3/§8.1 字面允许的一处显式偏离，需要评审确认并同步上层文档（处理方式
  类比 m4 D21）**：
  - 手册 §6.3/§8.1 允许"首次合同与 Architecture PR 同时提交，两个 Gate 都通过后合并"。
    但已合并实现里有三处把"一 PR 恰一 Gate"钉死：M1 `verify.ts` 的 label 冲突判定
    （存在**另一个** `gate:*` label → 对**任一** gate 的校验都返回 fail——已实机核对，
    双 label PR 会让 architecture 与 contract 两个方向的 `verifyGateApproval` 全部失败）；
    M2 hygiene 的"恰一个 `gate:*` label"（`gate-hygiene.ts` 实机核对，多 label 直接
    violation）；M2 D5/D6 的 label 语义与单 `gate:` 字段的 marker 格式。让双 Gate PR 成为
    一等公民 = 同时弱化三处已合并的防线 + 扩容 marker 格式，换来的只是省一个 PR。
  - **本文决定**：拆分。Architecture Gate PR（`architecture.md` + `projects.yaml`，
    **不含** `contracts/openapi.yaml`）先合并；Contract PR（`gate:contract`）随后提交，
    marker 的 `upstream_approvals` 引用已合并的 spec 与 architecture Gate（M2
    `UPSTREAM_GATES_FOR` 已备好 `contract: ['spec','architecture']`，§4 启用它）。
  - §6.3 的强制内核——"任何 openapi.yaml 新增或修改不得绕过合同检查"——不但没有被削弱，
    反而双保险：CI 侧 Contract Gate job 按路径触发、与 label 无关（谁改都会被检查）；
    hygiene 侧新增**互斥规则**（D9/§4.3）在 PR 阶段就拒绝"非 contract PR 夹带
    openapi.yaml"，而不是等它进了 main、M5 publish 找不到 provenance 才发现死角。
  - **需要同步的上层文档（评审通过时一并修订，不悄悄改语义）**：runbook §6.3 的 PR 文件
    清单与"一个 PR 同时包含两类文件时…"两句、§8.1"首次合同可以与 Architecture PR 同时
    提交…"一句，改写为拆分流程。`contracts/events.yaml` **不在互斥范围**（m4 D8 窄口径
    的一致延伸：events.yaml 没有 gate，仍可随 Architecture PR 提交，只触发 M4 的宽口径
    平台矩阵）。若评审不接受拆分，替代方案是扩容上述三处已合并机制，那是一次独立的
    重新设计，不在本文内偷做。
- **D9 — hygiene 新增全局互斥规则：任何 PR 的 changed files **以任一形式触及**
  `contracts/openapi.yaml`（`filename` 命中，**或 rename 条目的 `previous_filename`
  命中**）而 label 不是 `gate:contract` → violation**（含无 label 的普通实现 PR 与
  `gate:architecture` PR）。这条规则建立的不变量是 M5 的地基：**main 上每个
  `openapi.yaml` blob 都能唯一追溯到恰一个 `gate:contract` PR，且该 PR 的 changed files
  包含它**——这正是 `verifyGateApproval(gate='contract')` 的第 2/5 步（artifactPath ∈
  changed files + 本地 blob == 该 PR 版本）可满足的必要条件。没有它，任何实现 PR 顺手
  改一行合同就会让下一次 publish 无据可验（fail closed 但成为流程死角，只能靠再发一个
  "空转" Contract PR 解套）。
  **为什么必须同时看 `previous_filename`（自查抓到的 rename 绕过，m4 D19 同类）**：
  把 `contracts/openapi.yaml` **重命名走**（如改名为 `contracts/api.yaml`）的 PR，其
  changed files 只有一条 `status='renamed'`、`filename=新路径` 的条目——旧路径仅存在于
  `previous_filename` 字段。只匹配 `filename` 时：互斥规则错过它；M4 的
  `contract_changed`（按 m4 D8 精确匹配 `filename`）同样为 false，Contract Gate 不跑
  ——"重命名走"就成了唯一不经任何合同检查、效果等同删除合同的变更路径。修复后：
  rename-away 被互斥规则按 `previous_filename` 命中；即使在 `gate:contract` PR 内做
  rename-away，也会被 §4.2 的必需产物规则（新 `filename` ≠ `contracts/openapi.yaml`）
  拦下。**`contract_changed` 本身不为此加宽**——m4 D8 已明确它不决定删除类语义，
  删除/重命名走的拦截责任在本文的 hygiene 层，谁也不该回头去"修" M4 那个字段。
  **执行位置的现状修正（实机核对）**：hygiene 现状对无 gate label 的 PR **在读取
  changed files 之前就直接 pass**（`gate-hygiene.ts` 先查 label 再 fetch files）——
  互斥规则必须作为**先于三类分支（Gate 专属 / Scaffold 专属 / 纯通用）的通用前置检查**
  执行，代价是每个 PR 多一次全分页 files 读取（hygiene 本就逐 PR 运行，量级可接受）。
  同时**删除** `gate-hygiene.ts` 里 architecture 分支的 openapi.yaml operationId 校验段
  （实机核对第 214–243 行）：该路径被互斥规则整体禁止，保留会误导实现者以为
  architecture PR 允许携带合同；operationId 的内容校验归属 Contract Gate job（D6.2），
  `OPERATION_ID_RE` 常量保留并被两处共用。
- **D10 — 业务逻辑两层落位，镜像 hygiene/detect 的既有模式（m4 D10）**：CI 编排命令
  `sdd gate contract snapshot` / `sdd gate contract verify`（`cli/src/commands/gate/contract/`）+
  业务逻辑 `snapshotContractGateInputs` / `verifyContractGateSnapshot`
  （`factory/src/contract-gate.ts`）。workflow 只做工具安装、调用两个阶段并透传退出码；
  API 读取与 lint/diff/生成/编译的编排仍全部在 factory 内（可单测、可注入），不散落在
  YAML bash 里。外部工具（oasdiff、generator jar、gradle、swift）由 workflow 预装，
  verify 阶段运行前**逐个核对版本输出与内嵌期望值**，不符 →
  fail closed（防 runner 预装版本漂移——"钉死"必须在运行时可验证，否则只是愿望）。
- **D11 — M1 `verifyContractGateCheck` 需要一处加固补丁：反查并验证受信 Actions workflow
  identity，而不是仅按 name/app 判定**（对既有代码的精确修正，列入 §10.1 先行补丁表）：
  - **现状**（实机核对 `verify.ts:487-518`）：`checks.find(c => c.name === 'Contract
    Gate')` 取**第一个**同名匹配，不校验该 check 由谁产生，同名多条时行为取决于 API
    返回顺序。
  - **诚实的威胁分析（自查）**：required-check 机制已保证"改了 openapi.yaml 的 PR 能
    合并 ⇒ 真 Contract Gate 在最终 head 上 success"（D3 聚合 + `CI Gate` required），
    伪造同名 check **无法帮助合并**；evidence 端的剩余风险是**歧义**——产品 PR 可新增
    一个自带 workflow 产生同名 check（M2 D10 描述过的入口，required 侧已防、evidence
    侧未防），`find` 首个匹配的结果不确定。
  - **补丁（fail-closed 方向）**：`listForRef` 显式传 `filter:'all'`，收集同 head SHA 上
    全部同名 run；对每条候选由 `details_url`/check-suite 映射到 Actions workflow run，读取
    run metadata，要求 `head_sha` 相同、`event='pull_request'`，且 workflow identity 精确等于
    organization ruleset pin 的平台仓 `.github/workflows/ci-gate.yml@<trusted SHA>`。
    `app.id===GitHub Actions` 仅作为必要条件，不再被当作产生方身份证明（产品仓自带 workflow
    也有同一 app id）。在受信 identity 下按 `run_attempt`/完成时间选择最新 completed run：
    最新结论必须 success；旧的正常 re-run 不造成歧义。出现多个无法排序的受信候选、run
    metadata 缺失、identity 不可验证均 fail closed。实现开始前必须用隔离 org fixture 证明
    required-workflow check 能从 GitHub API 反查到上述 identity；若 API 不暴露足够身份，
    D11 阻塞实现并改用独立受信 GitHub App 产出 evidence，不允许退回 name+app.id。
- **D12 — `gate:contract` PR 必须有且仅有一个 `version:<v>` label 且与 marker.version
  一致**（hygiene 强制，对齐 m3 D26 对 architecture Gate 的同款收紧；通用 verifier 的
  "version label 可选"语义不变）。理由与 m3 D26 相同：M5 publish 将以
  `(approval, version, gate='contract')` 调用 `verifyGateApproval`，version 需要一个
  GitHub 原生、非 marker 自由文本的交叉核对点。contract PR 不触碰 `specs/**`，
  "version ↔ specs 路径段"检查对它天然空转，version label 是它唯一的版本外证。
  marker.version 语义 = 该合同服务的 spec 版本（如 `v1`）。
- **D13 — Contract Gate 的 breaking 判定以 merge-base 为 base，配合
  `strict_required_status_checks_policy` 关闭 TOCTOU 窗口（依赖 m3 §9.1 #4 补丁）**：
  - diff 的 base 侧取 **PR merge-base commit** 的 `contracts/` 子树（`GET
    /repos/{o}/{r}/compare/{base_ref}...{head_sha}` 响应的 `merge_base_commit.sha`，与
    PR files API 的 diff 语义一致），不是 base branch 的实时 tip——保证"这个 PR 把合同
    从什么改成什么"的比较对象与 reviewer 看到的 PR diff 完全相同。
  - **TOCTOU（自查——本文版本的"m3 D24 时刻"）**：PR 打开后、人工合并前，若**另一个**
    Contract PR 先合并改写了 main 上的合同，本 PR 的 merge-base 仍指旧版本，breaking
    判定参照过时基线——例如两个 PR 各自单看都非 breaking、叠加后删除了对方依赖的
    operation。修复不新造机制：m3 §9.1 #4 的 `strict_required_status_checks_policy`
    （"合并前分支必须最新"）已把这个窗口关死——main 前进 → 必须 Update branch → 产生
    新 head + 新 merge-base → `synchronize` 事件触发 Contract Gate 重跑，重跑用的就是
    新基线。该补丁因此从"M3 的 Scaffold 需要"升级为"M3 与 M4.5 共同的前置"（§10.1），
    M4.5 实现前必须确认已合并。
- **D14 — 产品仓 PR 模板的 marker 扩展只影响新建产品仓；存量仓靠规则而非模板**：
  `templates/monorepo-root` 的 `PULL_REQUEST_TEMPLATE/gate.md` 增加 contract 用的
  `migration_plan` 可选字段说明，并说明 breaking 时须提交 D7 的 migration 文件。模板是一次性脚手架
  （M2 D3/手册 §4.6），存量产品仓**不会自动获得新模板**——这不构成问题：hygiene 与
  Contract Gate 校验的都是 marker **内容**而非模板文件，存量仓作者手写字段即可；
  模板的定向同步属于 M8。`parseGateMarker` 的解析器按行前缀增量识别该字段，旧 marker
  无该行 → 字段为 undefined → 仅在检测到 breaking 时才要求存在（D7），完全向后兼容。
- **D15 — 不可信输入资源上限是 gate 契约，不留给 runner 默认值**：job
  `timeout-minutes: 45`；快照阶段限制 `contracts/` 总文件数 ≤500、总字节 ≤20 MiB、单文件
  ≤5 MiB，OpenAPI operation 数 ≤2000；verify 阶段每个 generator 最长 5 分钟、每语言
  build+test 最长 10 分钟、单语言生成目录 ≤500 MiB、捕获的单工具 stdout/stderr 各 ≤1 MiB。
  任一超限立即终止对应进程树并 fail closed，报告写明命中的 limit；具体阈值未来可经方案
  评审调整，但实现不得省略 timeout/size/process-tree kill 机制。

## 1. `ci-gate.yml` 的 Contract Gate job（在 M4 版之上的增量）

本节 YAML 是对 **m4-foundations §2.9/§3.1 规划的 M4 版 `ci-gate.yml`** 的增量（不是对
当前 M2 stub 的——M2 stub 会先被 M4 重写，见文首实现顺序约束）。

```yaml
jobs:
  detect:
    # M4 版不变；contract_changed 输出 m4 已接线（11 个输出之一）

  backend: / web: / ios: / android:
    # M4 版不变

  contract:
    name: Contract Gate            # D1：job name = check context，冻结；守卫测试保护
    needs: [detect]
    if: needs.detect.outputs.contract_changed == 'true'
    runs-on: ubuntu-latest         # D5/待决 #1：swift-linux 验证失败则整 job 换 macos-14
    timeout-minutes: 45            # D15：不可信生成/编译的 job 总上限
    steps:
      - name: resolve trusted workflow source     # 逐字复用 pr-hygiene/detect 的既有模式
        id: workflow-source
        # ...
      - name: checkout platform repo               # 只 checkout 平台仓自身 pinned SHA
        uses: actions/checkout@<pinned-sha>
        with: { repository: <平台仓>, ref: "${{ github.workflow_sha }}", path: platform }
      - name: setup node + pnpm install --frozen-lockfile + build
        # ...（与 pr-hygiene/detect 相同）
      - name: setup java (temurin 21)              # generator jar 运行 + kotlin 编译共用
        uses: actions/setup-java@<pinned-sha>
        with: { distribution: temurin, java-version: '21' }
      - name: install pinned oasdiff               # 固定版本二进制 + sha256 校验后入 PATH
      - name: install pinned openapi-generator-cli # 固定版本 jar + sha256 校验
      - name: install pinned gradle                # kotlin 编译（D5）
      - name: install pinned swift toolchain       # swift 编译（D5/待决 #1）
      - name: snapshot contract inputs
        working-directory: platform
        env: { GITHUB_TOKEN: "${{ secrets.GITHUB_TOKEN }}" }
        run: |
          node cli/bin/run.js gate contract snapshot \
            --repo "${{ github.event.pull_request.base.repo.full_name }}" \
            --pr "${{ github.event.pull_request.number }}" \
            --out "${{ runner.temp }}/contract-snapshot"
      - name: verify contract snapshot
        working-directory: platform
        # 刻意不设置 GITHUB_TOKEN；ToolRunner 再做环境 allowlist（D4）
        run: |
          node cli/bin/run.js gate contract verify \
            --snapshot "${{ runner.temp }}/contract-snapshot"

  CI Gate:
    needs: [detect, backend, web, ios, android, contract]   # 追加 contract
    if: always()
    # 聚合逻辑见 §3
```

- 不 checkout 产品仓（D4）；`base.repo.full_name` 沿用"永远不用 `head.repo`"的既有原则。
- 工具安装步骤全部带版本 + 校验（D5/D10）：下载类步骤（oasdiff/jar/swift tarball）内联
  `sha256sum -c`；action 类（checkout/setup-java/setup-gradle）按 M2 D10 惯例固定完整
  commit SHA。安装的具体版本号集中声明在 workflow 顶部 `env:` 块，便于与
  `verifyContractGateSnapshot` 内嵌期望值（§2.2）比对时只有一处来源可漂移——漂移会被运行时版本
  核对当场抓住。
- `contract_changed=false` 的 PR：job `skipped`，零成本；这覆盖绝大多数 PR（合同变更
  在产品生命周期里是低频事件），Contract Gate 的工具链安装成本只在真正需要时支付。

## 2. `sdd gate contract` / Contract Gate 业务逻辑

### 2.1 CLI 接口

```bash
sdd gate contract snapshot --repo <owner/name> --pr <number> --out <dir>
sdd gate contract verify --snapshot <dir>
```

- 两阶段均为 CI 专用：`snapshot` 是唯一 API-backed 阶段；`verify` 只读本地快照且拒绝
  检测到任何凭据环境变量。拆分是 D4 的凭据隔离边界，不提供合并两阶段的便捷模式。
- 输出：结构化 JSON 报告到 stdout（§2.5），Actions 日志里直接可读。
- 退出码：`0` = 全部检查通过（含"breaking 已确认"的放行）；`3` = 任一检查失败或无法
  判定（fail closed，含工具版本不符、API 读取失败、`openapi.yaml` 解析失败）；`2` =
  输入错误（参数缺失/非法）。

### 2.2 检查流水（snapshot 完成后才进入 tokenless verify；verify 除工具可信性/资源超限外
fail 不短路，尽量一次报告全部问题）

```text
snapshot（唯一持有 GITHUB_TOKEN 的阶段）：
1. fetchPullRequest（复用 m4 D16 的共享 helper）：base repo、head SHA、labels、body、
   changed_files 计数。
2. fetchChangedFiles（全量分页 + changed_files 计数校验，m4 D22 契约原样复用）。
   独立复算 contract_changed；为 false → 退出 3。
3. 计算 merge-base（D13）。
4. 读取 head/merge-base 两侧 contracts/ 子树；执行 Contents API 1000 条完整性守卫与
   D15 文件数/字节上限；把字节、PR 元数据、changed files、head/merge-base SHA 写入
   权限仅当前用户可读的快照目录。step 结束后 GITHUB_TOKEN 不进入后续环境。

verify（无 token、无 API 调用）：
0. 拒绝凭据环境变量；工具版本核对（D10）：oasdiff --version / generator jar 版本 / gradle --version /
   swift --version 逐个与内嵌期望值精确比对；任一不符 → 立即退出 3（这一条短路：
   工具不可信则后续结果全不可信）。
1. 校验并读取快照；head SHA 侧 → <tmp>/head/contracts/**；merge-base
   侧 → <tmp>/base/contracts/**（base 侧无 openapi.yaml → 记为"首次引入"）。读取方式：
   Contents API 按目录逐层 walk（contracts/ 子树小且深度有限；不用 recursive tree，
   避免 truncated 分支）。任何非 404 的读取失败 → 退出 3。**完整性守卫（m4 D22 同款
   "证明读完了"信号）**：Contents API 对单目录最多返回 1000 条且不分页、不报截断——
   任一目录的返回条目数达到该上限 → 无法证明读全 → 退出 3，不拿可能不完整的子树继续
   往下检查（漏读的文件若被 $ref 引用会让 lint 失败——那是碰巧安全；不碰巧的情况是
   漏读与检查无关但存在性本身可疑的文件，统一 fail closed 不赌）。
5. openapi 版本检查（D6.1 后半）：head 侧 openapi.yaml 的 `openapi` 字段匹配 ^3\.0\.\d+$。
6. $ref 安全扫描（D6.3，解析器跑之前先扫）：遍历全部 $ref 字符串值——允许 `#/...`；
   允许相对路径且 POSIX normalize 后仍在 contracts/ 内；其余（URL scheme、绝对路径、
   逃逸 ..）→ violation。
7. spectral lint（D6.1/D6.2/D6.4）：extends spectral:oas + 平台自定义规则（operationId
   必有/唯一/匹配 OPERATION_ID_RE；requestBody 与 2xx response 的每个 media type 至少
   一个 example/examples）。resolver 限定本地文件（步骤 6 已保证无远程引用）。error 级
   发现 → violations。
8. breaking diff（D7）：非首次引入时，oasdiff breaking <base> <head> --format json；
   ERR 级发现非空 → 按 D7 校验 marker 指向的 migration 文件路径、changed-file 状态、
   head 内容及 `sdd:breaking-change` 机读确认；全部成立才放行，否则 violations。
   base 侧存在但解析失败/版本不受支持（遗留合同）→ 视同"未知 breaking"走同一条 ack
   判定（D7）。无 breaking 但 marker 指向 migration plan → warning（悬空计划，不 fail）。
9. 三语言生成 + 编译 + 最小测试（D6.6）：对 (typescript-fetch, swift5, kotlin) 各：
   java -jar openapi-generator-cli generate -c <平台仓 config> -i <tmp>/head/contracts/
   openapi.yaml -o <tmp>/gen/<lang> → 注入平台 harness 测试文件 → 编译 + 测试
   （tsc+vitest / gradle build test / swift build && swift test）。任一非零 → violations
   （报告含该语言的工具 stderr 摘要）；每个子进程应用 D4 环境 allowlist 与 D15 的
   timeout/output limit，超限时终止整个进程树。
10. 汇总：violations 为空 → 打印报告、退出 0；否则打印报告（含全部 violations +
    warnings）、退出 3。
```

### 2.3 spectral 规则文件（平台仓资产）

`factory/contract-rules/spectral.yaml`（随 `@sdd/factory` 打包分发，verify 阶段
从包内路径加载，**不接受产品仓提供的规则文件**——规则是平台的裁量标准，产品 PR 不可改）：

```yaml
extends: spectral:oas
rules:
  # spectral:oas 内置已覆盖 operation-operationId / operation-operationId-unique /
  # oas3-unused-component / no-$ref-siblings 等；以下为 M4.5 自定义收紧
  sdd-operation-id-pattern:      # D6.2：与 M2 OPERATION_ID_RE 同一正则
    severity: error
    given: "$.paths[*][*].operationId"
    then: { function: pattern, functionOptions: { match: "^[a-z][a-zA-Z0-9_-]*$" } }
  sdd-request-body-example:      # D6.4
    severity: error
    given: "$.paths[*][*].requestBody.content[*]"
    then: { function: schema, functionOptions: { schema: { anyOf: [{ required: [example] }, { required: [examples] }] } } }
  sdd-2xx-response-example:      # D6.4
    severity: error
    given: "$.paths[*][*].responses[?(@property.match(/^2\\d\\d$/))].content[*]"

```

## 3. `CI Gate` 聚合扩展（D3 落地）

在 m4 §3.1 聚合 step 的四段平台判断之后（detect-first 短路检查仍然是第一条可执行语句，
m4 D13 不变）追加：

```bash
        # contract（M4.5，D3——键控 contract_changed，不是平台布尔）
        if [ "${{ needs.contract.result }}" = "failure" ] || [ "${{ needs.contract.result }}" = "cancelled" ]; then
          echo "Contract Gate failed"; FAILED=1
        elif [ "${{ needs.detect.outputs.contract_changed }}" = "true" ] && [ "${{ needs.contract.result }}" != "success" ]; then
          echo "Contract Gate was required but did not succeed (result: ${{ needs.contract.result }})"; FAILED=1
        fi
```

- 与四段平台判断同风格的直白展开（m4 §3.1 已论证为什么不用循环）。
- **自查（相似字段不混用）**：这里读的是 `contract_changed`，**不是** `contracts/**`
  宽口径信号——只改 `events.yaml` 的 PR 会触发全平台矩阵（M4 宽口径）但 Contract Gate
  正确保持 `skipped` 且聚合通过（m4 §6 已有该测试用例，本文在 §7 加聚合侧的对应断言）。
- ruleset 不新增 required status check：`Contract Gate` 的强制力经由"`CI Gate` 是
  required check + 聚合规则"传导，M1 provenance 另行按名字读取它的 check run 作为
  evidence——两个消费路径都不要求它自身成为 required context（保持 M2 required 集合
  `CI Gate` + `PR hygiene` 不变，避免又一轮 required-check 自举）。

**evidence 闭环论证（M5 消费的地基，自查过重放场景）**：合同 PR 能合并 ⇒ `CI Gate` 在
最终 head 上 success ⇒（该 PR 改了 openapi.yaml ⇒ `contract_changed=true`）⇒ 聚合规则
要求 `Contract Gate` 在**同一次 run**（同一 head SHA）success；`strict_required_status_
checks_policy`（§10.1 #3）保证 main 前进后必须 Update branch 重跑，check 不会停留在旧
head；ruleset 的 stale review dismissal 保证 approval 也绑定最终 head。于是 M1
`verifyGateApproval(gate='contract')` 要求的"approved head SHA 上存在 success 的
`Contract Gate`"对任何合法合并的合同 PR 恒可满足，对绕过路径（label 冒充、旧 SHA 的
成功、无 label 的夹带合并）恒不可满足（依次被 D11 补丁 + M1 head-SHA 绑定 + D9 互斥
规则挡住）。

## 4. `PR hygiene` 扩展（D8/D9/D12 落地，`factory/src/gate-hygiene.ts`）

### 4.1 通用前置检查（新增，先于 Gate/Scaffold/通用三类分支）

任何 PR（含无 gate label 的）：changed files 含 `contracts/openapi.yaml` 且 PR 的 gate
label 不是恰好 `gate:contract` → violation
`"contracts/openapi.yaml may only change in a gate:contract PR"`。实现注意（实机核对
现状）：现有代码对无 gate label 的 PR 在 fetch changed files **之前**就 return pass——
本规则要求把 changed files 读取提到 label 分支之前（或在无 label 分支里补读），全分页 +
`changed_files` 计数校验沿用 m4 D22 修正后的共享 helper。

### 4.2 `gate:contract` 分支补全（M2 半状态 → 完整规则）

对 `gate:contract` PR，在既有通用 Gate 规则（恰一 label、marker 匹配、version 格式）
之上：

1. **必需产物**：`getRequiredArtifacts('contract')` 从 `[]` 改为
   `['contracts/openapi.yaml']`（实机核对：现状 default 分支返回空数组，即 M2 的
   contract PR 连"必须真的改了合同"都不查）。
2. **上游批准**：§3.5.4 的执行条件从 `['architecture','design','plan']` 扩为含
   `'contract'`——`UPSTREAM_GATES_FOR.contract = ['spec','architecture']` 这行死数据
   随之激活：marker 必须给出 `upstream_approvals.spec` 与 `.architecture`，两者都要
   重验证（存在、已合并 main、带正确 `gate:*` label——复用既有防伪逻辑，零新代码）。
   这同时把 D8 的拆分顺序（Architecture 先合并、Contract 后行）变成机检约束。
3. **version label**（D12）：有且仅有一个 `version:<v>` label 且等于 marker.version；
   缺失/重复/不一致 → violation。
4. **禁止整文件删除**（D7 末条）：changed files 里 `contracts/openapi.yaml` 的
   `status === 'removed'` → violation `"deleting the contract file has no supported
   gate path"`。
5. **删除** architecture 分支的 openapi.yaml operationId 校验段（D9 末条；该段的
   触发条件已被 4.1 的互斥规则判为 violation，保留即死代码 + 误导）。

hygiene **不**重复 Contract Gate job 的内容检查（lint/diff/生成编译），也**不**检查
"Contract Gate check 已 success"——后者与 hygiene 并行运行存在天然竞态，强制力由 §3 的
聚合规则承担（M2 §3.5 末尾那条"contract 专属规则随 M4.5 落地"的预告，落成的就是本节 +
§3 的组合，其中"check success"部分落在聚合而非 hygiene，特此对 M2 文档说明作精确化）。

### 4.3 marker 解析扩展（D14）

`parseGateMarker` 增加 `migration_plan:` 可选字段的行解析；仅接受 D7 的规范化仓内路径，
不得把任意非空字符串当作合法引用。旧 marker 无该行 → undefined → 只在 Contract Gate
检测到 breaking 时才要求。`templates/monorepo-root` 的
`PULL_REQUEST_TEMPLATE/gate.md` 同步加注释说明该字段与 migration 文件机读块（模板变更只惠及新产品仓，
存量仓不回填——D14）。
