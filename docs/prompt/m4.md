## 阶段 2：`sdd impact`

### 2.1 Reader 与接口

在 `factory/src/impact.ts` 实现：

- `ChangedPath { path, status, previousPath? }`；`previousPath` 仅 rename 使用。
- `ImpactReader.listChangedPaths(base, head)` 与 `readFileAt(ref, path)`；`null` 只表示该 ref 上文件不存在。
- API reader：compare API + Contents API；必须确认文件列表完整，否则抛错。此路径是独立 `sdd impact --repo` 预览，不是 CI Gate 权威路径。
- local reader：`git diff --name-status -M`，正确解析 `R<score>\t<old>\t<new>`；`git show ref:path` 区分“不存在”和真正错误。
- `computeImpact({ reader, base, head, changedPaths? })`：detect 恒传入已完整验证的 PR-files 列表；传入时禁止再次调用 `listChangedPaths`。

### 2.2 平台布尔

严格按 §2.4/§2.5：

- contracts：全部 existing/declared 语义范围的平台。
- design：web/ios/android，不含 backend。
- architecture/plan/未知 specs 文件：全部平台。
- spec.md：整篇规范化文本相同才不触发；任何实质变化保守全平台。
- rename 比较 base 内容时使用 `previousPath`。
- 平台布尔绝不能由 ID semantic diff 驱动；两套算法回答不同问题。

### 2.3 D20 报表字段：三种算法，禁止通用锚点

实现三个独立的小函数，均返回互斥的 `{ added, removed, changed }`：

1. `diffRequirementSections`：
   - 只认精确 canonical heading：`^### (REQ-[A-Z0-9]+-\d+)\s*$`。
   - section 到下一个同级/更高级 heading 或 EOF。
   - 表格/依赖/追踪区里的重复 REQ 引用不是 declaration。
2. `diffDesignScreens`：
   - 只认 Design 模板 §2 “屏幕清单”表中的 Screen ID 行。
   - 同一 SCR 在 §8 页面↔OpenAPI 映射再次出现不能形成第二个对象。
   - 先比较 canonical 表行；去掉该表后的正文若有实质变化且无法归属具体 screen，把 base/head canonical screen ID 并集全部加入 changed，保守多报而不漏报。
3. `diffOpenApiOperations`：
   - 用 factory 已有 `yaml` 依赖解析 `contracts/openapi.yaml`。
   - 遍历 `paths.<path>.<标准 HTTP method>`，每个 operation 必须有合法且全局唯一的 `operationId`。
   - 按 operationId 对完整 operation object 做 key-sorted stable structural comparison；YAML key 顺序变化不得产生假变化。
   - `summary`/parameters 位于 `operationId` 前、responses 位于其后都必须属于同一个 operation。
   - YAML 解析失败、非法/重复 operationId fail closed，不退回文本猜测。

`changed.requirements/screens/operations = added ∪ removed ∪ changed`；`breaking` 只取 operation `removed`，不要把 modified operation 当成 M4.5 的完整 breaking diff。

### 2.4 CLI

新增 `cli/src/commands/impact.ts`：

- `sdd impact --base --head [--repo owner/name] [--format json|text]`。
- `--repo` 模式要求两个 ref 都是完整 40-hex SHA 且需要 `GITHUB_TOKEN`。
- 无 `--repo` 为本地 git 预览模式。
- 默认 JSON；text 是同一 model 的人类可读 renderer。
- 退出码：0 成功、2 输入错误、3 分析/读取/校验失败。
- 输出必须通过 `validateImpactDocument`；复用 `SDDImpact` 类型，不另造同形类型。

登记 CLI tsup entry、命令导出和 package build discovery，不能只新增源码文件。

## 阶段 3：`sdd gate detect`

在 `factory/src/detect.ts` 和 `cli/src/commands/gate/detect.ts` 实现 §2.2 的完整 10 步：

1. 一次读取 PR 元数据，取得 base/head SHA、labels、repo 与 `changed_files`。
2. 分别读取并校验 base/head `projects.yaml`。任一读取/校验失败即 fail closed。
3. 构造 baseComponents/headComponents。
4. 对每个 **head 声明** component 做 base/head tree 存在性检查：
   - 优先 recursive Git tree；`truncated=true` 时逐 component Contents API fallback。
   - base=false/head=false：尚未 scaffold，允许，`existing=false`。
   - base=false/head=true、base=true/head=true：正常，head 存在的 path 进入对应 `*_paths`。
   - base=true/head=false 且仍在 head 声明：detect 立即失败。单/多 component 平台都不能跳过。
5. PR-files 全分页并与 `changed_files` 计数核对；不等即失败。
6. 路径分类使用带 `/` 边界的前缀匹配；rename 的 `path` 对 headComponents，`previousPath` 对 baseComponents。
7. `specs/**`、`design/**`、`contracts/**` 任一命中时，同进程调用 `computeImpact`，传入步骤 5 的 `changedPaths`；失败/非法输出直接传播。
8. `platform:*` label 只能 OR false→true。
9. 最终 `AND existing[platform]`；`*_paths` 必须与同一次存在性检查一致。
10. 独立计算 `contract_changed`：仅 `contracts/openapi.yaml` 的 added/modified/rename-in，不把整个 `contracts/**` bucket 混进来；按规格处理 removed。

输出固定 JSON：四平台布尔、四个 `*_paths` 数组、`contract_changed`、`product_repo`、`head_sha`。CLI 恒 API-backed，退出码 0/2/3。

## 阶段 4：四个平台 reusable workflow

新增平台仓 `.github/workflows/{java,web,ios,android}.yml`，均为 `workflow_call`：

- inputs：`product_repo`、`head_sha`、JSON string `paths`。
- checkout 显式使用 input repo/ref；token 方案必须与阶段 0 spike 结论一致。
- matrix 使用 `fromJSON(inputs.paths)`；D18/D25 已保证外层 detected=true 时 paths 非空，不依赖空 matrix 语义保正确。
- 每个平台按 M3 最终命令契约执行 lint/typecheck-equivalent/test/build。
- 显式 setup 工具链并核对版本：Java 21/Gradle wrapper、Node 24+Corepack/pnpm、Xcode+Tuist+SwiftLint、Android JDK+SDK/build-tools。
- 不读取发布/签名 secret，不新增写权限，不做 release。

不要把四份 workflow 泛化成“通用命令参数”；平台工具链差异是真实的。

## 阶段 5：扩展 `ci-gate.yml`

1. 触发器增加 `labeled`/`unlabeled`，保证 `platform:*` 标签变化会重跑。
2. `detect` job 复用 `pr-hygiene.yml` 已验证的可信平台源码解析/checkout/build 模式；只 checkout pinned 平台仓，不 checkout 产品 PR。
3. 给 detect step 设置 `id: detect`；job outputs 的 11 个字段全部引用 `${{ steps.detect.outputs.* }}`，禁止字面量或部分断连。
4. 新增 backend/web/ios/android 四个外层 caller job：`needs: detect`，外层 `if: needs.detect.outputs.<platform> == 'true'`，相对路径调用同 commit reusable workflow并显式传 inputs。iOS 的 if 必须在外层 caller，未检测到时不分配 macOS runner。
5. `CI Gate`：`needs: [detect, backend, web, ios, android]`，`if: always()`，job name/context 保持精确的 `CI Gate`。
6. 聚合 step 第一条可执行判断必须是 `needs.detect.result != success → exit 1`。随后对每个平台应用：
   - result failure/cancelled：无条件失败。
   - detected=true 且 result=skipped：失败。
   - 其它允许状态按 §3 真值表处理。

## 测试

逐条落实 `docs/sdd/m4-foundations.md` §6，不要只写文档断言。最低必须覆盖：

- YAML outputs 接线：11 个 outputs 都引用真实存在的 detect step id。
- 路径边界、同平台多 component、未匹配 apps、rename old/new 分别对 base/head components。
- D25 base/head 存在性四种组合；单 component 删除和多 component 删除都 fail closed；tree truncation fallback。
- D22 `changed_files=5000` 但只抓到 3000 → fail closed；正常计数相等通过；hygiene 回归。
- computeImpact 有预取 paths 时 reader 的 `listChangedPaths` 调用次数为 0。
- spec 全文 diff：无 ID 段落实质变化仍触发全部平台；纯规范化噪音不触发。
- requirement canonical heading、重复引用不成为第二锚点。
- screen 清单与 §8 重复引用；清单外正文变化保守报告全部 canonical screen。
- OpenAPI：`operationId` 前后的字段变化都报告正确 operation；key reorder 不报告；非法/重复 ID/YAML 失败 fail closed。
- breaking 仅 removed operationId。
- `contract_changed` 与 contracts 宽平台信号分离。
- `computeImpact` 失败传播到 detect/退出码 3。
- CI Gate 5 行标准真值表 + 2 个异常状态；detect failure + 全平台 skipped 必须失败且检查顺序正确。
- GitHub Actions 实测：backend-only 不调度 macOS；多 component matrix；M3 Scaffold PR 正确路由；四平台工具链真实冒烟。
- D24 token spike 与既有产品仓 ruleset pin 迁移 E2E；无法执行的 E2E 必须逐项如实标记未验证及原因。
