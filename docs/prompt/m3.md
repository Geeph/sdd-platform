### 阶段 0（先做）：§9.1 的四处前置补丁

这些改动小、独立、不依赖后面任何阶段，先做完能让 provenance/schema 层先稳定下来。

1. `schemas/projects.schema.json`：`components[].template_ref` 加 `"pattern": "^[0-9a-f]{40}$"`（D19）；`components[].owner` 加 `"pattern": "^[a-z][a-z0-9-]*$"`（D23，与 `id`/`product` 同风格）。跑 `pnpm --filter @sdd/schemas test` 确认既有 fixture 不受影响（如果某个 fixture 用的是非 SHA 的 `template_ref`，改成 40-hex 占位值）。
2. `provenance/src/verify.ts`：label 一致性检查前加 `gate:<gate>` **存在性**断言（D22，§3.5 给出了精确的 diff 意图，照抄即可）；`version:<v>` 保持"若存在须一致，不要求存在"不变。`provenance/test/verify.test.ts` 新增回归用例：PR 完全不带任何 `gate:*`/`version:*` label（既不冲突也不存在）→ 断言 `ok:false`（这条测试在修复前应该失败，修复后应该通过，用来证明确实堵上了这个洞）。
3. `factory/src/github-write.ts` 的 `reconcileRepositoryRuleset`：**当前代码在 `required_status_checks` 规则参数里写的字段名是 `strict_required_status_checks`（约第 1077 行，`desiredRules.push({ type: 'required_status_checks', parameters: { ..., strict_required_status_checks: true } })`）。这是一个字段名拼写问题，不是缺失新功能**——已经用 GitHub 官方 OpenAPI schema（`github/rest-api-description` 仓库）核实过，正确字段名是 `strict_required_status_checks_policy`，而且这个字段在 schema 里被标为该 parameters 对象的 **required** 属性；当前拼写等于完全没提供这个必需字段，实际发给 GitHub 的请求很可能因为缺少必需属性被拒绝，或者这个属性被忽略、`required_status_checks` 规则从未真正启用"分支必须最新"这条语义。**把这一个字符串改对**（`strict_required_status_checks` → `strict_required_status_checks_policy`），不要动其它逻辑。改完后如果 `github-write.ts` 或其测试里有 mock 断言检查这个字段名，一并更新。这是 D24 依赖的基础设施（详见阶段 4）。

### 阶段 2：scaffold 只读侧（dry-run）

按 §2.1–§2.3、§2.4、§3.1–§3.3、§3.6 实现：

- `factory/src/scaffold/{types,plan,render,subtree}.ts`：`compileScaffoldPlan`、`renderComponent`、**`verifyComponentSubtree`（D25）——这是本轮最容易写错的部分，逐字看 §0 D25 的四个步骤**：(1) 递归读 tree，只保留目标 `path` 前缀下的条目，`path` 之外一律忽略；(2) 路径集合必须与期望文件集完全相等（缺文件、多文件都算失败）；(3) **对每个匹配路径，用 blob SHA 调用 `git/blobs/{sha}` 取回实际内容（base64 解码），计算 `sha256(content)`，与期望的 `output_sha256` 比较——绝对不要直接把 tree 条目自带的 blob SHA 拿去和 `output_sha256` 比较**（Git 的 blob SHA 是 `sha1("blob "+字节数+"\0"+内容)`，`output_sha256` 是 `sha256(内容)`，是两个完全不同的哈希函数、不同输入，这两个值不会因为内容相同就相等——这正是第 3 稿评审抓到的 bug，写完后自己造一个"内容相同"的测试用例验证真的判定为通过，而不是恰好蒙对）。
- `cli/src/git-reader.ts`：`createLocalGitReader(repoRoot)`，`@sdd/provenance` 的 `GitReader` 接口首个真实实现（D8）。参考 `cli/src/local-reader.ts` 的 git shell-out 风格，但注意两者方向相反——`local-reader.ts` 读**平台仓**模板字节（`GitHubReadPort`），这里读**产品仓**本地 worktree（`GitReader`），不要混用或试图合并成一个文件（可以抽取共享的"找 git 仓根/读 remote owner-repo"小工具，但两个接口本身要分开）。
- `cli/src/commands/product/scaffold.ts`：新命令，参照 `cli/src/commands/product/init.ts` 的参数解析/校验/文本渲染风格。按 §2.1 实现 flags（**没有 `--platform-ref`**，每个 component 的 `template_ref` 已经是 pin 住的 commit，直接用；`--architecture-pr`/`--architecture-merge-sha` 二选一 + `--architecture-version`）。
- Preflight 严格按 §2.3"Preflight"编号的 9 步实现，**第 3 步的 main 新鲜度校验（D18）不能跳**：读 main 当前 HEAD 的 `projects.yaml` blob，要求与本地 worktree 一致，这是防止"引用一个已被后续 Architecture Gate 取代的旧 PR 号"的关键防线，跟第 2 步的 `verifyGateApproval` 是两回事、缺一不可。
- dry-run JSON 契约按 §2.2：`authorization.main_fresh` 字段、`template_source` 不再有 `requested_ref`（因为 D19 之后它和 `resolved_commit` 恒等）、`operation_id` 只由待生成 component 参与哈希。canonical JSON、byte-identical、零 GitHub 写——这几条 M2 已经验证过同一套约束，照搬即可。

### 阶段 3：scaffold 写侧（真实生成）

按 §2.3"生成"编号的 9 步 + §0 D20 实现：

- `factory/src/scaffold/publish.ts`：`publishComponentBranch` + `upsertScaffoldPull`。**`publishComponentBranch` 绝不能复用 `publishSnapshot` 本体**——`github-write.ts` 里 `publishSnapshot`（约第 464 行）和 `init.ts` 的 `applyInitPlan`（约第 104 行）都显式禁止 `apps/*` 路径，这是 M2 故意的安全设计（防止 `product init` 意外写平台代码），scaffold 需要的写操作恰恰相反（只允许 pending components 的 `path` 前缀）。写一个新函数，Git Data blob→tree→commit 的机制可以参考 `publishSnapshot`/`upsertBootstrapPull` 的结构，但**路径校验逻辑要反过来写，不要尝试"复用+绕过"已有的禁止逻辑**。
- 分支/PR 复用前必须先跑 D25 的 `verifyComponentSubtree`：查到候选 PR 后**显式核对** `base.repo`/`base.ref==='main'`/`head.repo`/`head.ref` 与目标一致（不要只信任 `head` query 参数本身），用这次响应自带的 `head.sha` 做后续校验（不要再单独调用一次 `git/ref/heads/{branch}`，两次读之间会产生竞态）。
- D20 的分流逻辑：内容校验通过 + 找到 `open` PR → 复用；内容校验通过 + 分支存在但无 PR → 补建 PR（**不重建分支**，这是"createRef 成功、PR 创建前崩溃"的正常恢复路径，之前一稿在这里有自相矛盾，已经在 §0 D20 修好了，照着实现就行）；内容校验不通过 → `conflict`，零写。

### 阶段 4：merge-time 重新校验（D24）

`factory/src/gate-hygiene.ts` 新增 Scaffold PR 专属规则分支（识别依据：changed files 里有新增的 `apps/**/template.lock`，且无 `gate:*` label）。**两层都要实现，只做第一层是不够的**：

1. 第一层：读新增 lock 文件内容，解析 `component`/`template`/`template_ref`，与**当前 main**（不是 PR 分支自己的树）的 `projects.yaml` 比对，四者都要匹配。
2. 第二层：**不能信任第一层读到的 lock 文件自己声明的 `output_sha256`**——用第一层已经从当前 main 读到的 `component`/`template_ref` 信息，调用 `resolveCommit`/`readTemplateTree`/`renderComponent` 对平台仓在 `template_ref` 这个历史 commit 上独立重新渲染，得到一份全新算出的期望文件集，再对 **PR head** tree 调用 D25 的 `verifyComponentSubtree` 比对。如果只做"读 lock 文件说什么、和 PR 里实际文件比"，攻击者同时改应用文件和 lock 文件的 `output_sha256` 就能绕过——这正是第 3 稿评审抓到的具体攻击场景，务必按"独立重新渲染"实现，不要走捷径读 PR 自己的 lock 文件当作期望值。

这一层读平台仓走 API（`resolveCommit`/`readTemplateTree`），**不是** `pr-hygiene.yml` workflow 自己 checkout 出来跑 `sdd` 用的那份本地文件（那份 pin 死了一个固定 SHA，跟某个 component 具体的 `template_ref` 无关）。
